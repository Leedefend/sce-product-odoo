# 自定义前端现状专项审计 v1

审计范围：`sc_frontend_acceptance`，finance 与 project A member，桌面 1440×900、1280×800，窄屏 390×844。页面事实来自当前 `AppShell`、`HomeView`、`MyWorkView`、`ActionView` 与真实验收入口。

## 结论

- P0：0。登录、角色隔离、公司切换和付款列表真实入口可用。
- P1：3。付款列表标题/上下文/主操作层级不够稳定；导航与业务标题视觉权重接近；异常状态缺少统一页面级结构。
- P2：7。间距、圆角、阴影、表格密度和状态标签存在多套写法；公司/项目上下文在不同页面的展示位置不完全一致；筛选工具栏在 1280 宽度容易拥挤；辅助文案中仍有技术化表述。
- P3：4。移动端完整办理、键盘导航细节、主题深度优化、页面级组件目录可后续迭代。

## 观察与决策

| 区域 | 当前事实 | 级别 | 后续规则 |
| --- | --- | --- | --- |
| AppShell | 已展示公司、项目、角色和菜单，但上下文控件占用侧栏较多 | P1 | 顶栏保留公司与用户，侧栏保留项目范围；业务标题区不重复堆叠 |
| 导航 | 权威 release/delivery nav 驱动，空导航保持安全空态 | P1 | 不恢复 `result.nav`；一级、二级层级和当前态统一 |
| ActionView | 契约驱动筛选、表格、状态和行操作 | P1 | 统一 PageHeader、FilterBar、DataTable shell；每页一个主操作 |
| 筛选 | 快速筛选、保存筛选、分组等能力较多 | P2 | 默认展示高频筛选，其余进入“更多筛选” |
| 表格 | 金额、状态和行操作已有字段来源，但密度规则不统一 | P2 | 金额右对齐、状态使用语义标签、行高 44px |
| 状态 | 已有加载/空/错误/权限分支 | P2 | 统一 LoadingState、EmptyState、ErrorState、PermissionState 文案 |
| 响应式 | 桌面可用，窄屏主要依靠换行和滚动 | P2 | 1280 不遮挡主操作；390 仅保证查看、待办和轻量操作 |
| 文案 | 普通用户页面以中文为主，少量技术状态会泄漏 | P2 | 技术详情默认隐藏，仅支持人员展开 |

## 验收事实

- finance 公司 A 仅显示 FE-A、FE-B；切换公司 B 后仅显示 FE-C。
- project A member 仅显示 FE Project A，访问 B/C 记录得到权限拒绝。
- 付款列表真实入口为动态 action/menu，不依赖跨数据库硬编码 ID。
- 当前页面未发现阻断性的 console/page error；后续回归继续保存 report.json 与截图。

## FE-P01 追加事实

- 当前壳层已经有侧栏公司/项目/角色上下文，顶部原先主要承担面包屑、标题和操作，导致业务标题与上下文视觉权重接近；本轮在顶部增加紧凑上下文摘要，窄屏自动隐藏，侧栏仍保留完整切换控件。
- 角色首页已有欢迎区、今日待办、风险、指标和辅助入口等契约区域；没有真实数据时由契约状态控制，不新增硬编码经营数字。
- 初始化加载、初始化失败和权威空导航均已有 `StatusPanel` 分支；后续统一中文标题、说明、重试和返回路径，技术 trace 继续默认隐藏。
- 1440/1280 桌面布局保持内容区和主操作可见；390 窄屏优先保留查看与侧栏关闭能力，不承诺完整业务办理。
