## 2026-06-12 结论修订

菜单整合必须先按产品域、事实模型和用户意图判定，不能按旧菜单分组硬套，也不能把多个旧入口简单压成一个“综合办理”页面。

本轮确认的产品化判断顺序：

1. 先用业务分类字典和模型事实识别产品域，例如 `contract`、`finance`、`invoice_tax`、`material`、`site`，后续补齐 `labor`、`equipment`、`subcontract`、`hr_admin`、`document` 等域。
2. 再用旧菜单背后的 `res_model`、业务分类、历史数据来源判断事实模型和生命周期。
3. 再按用户意图区分办理、查询、分析、配置、主数据、来源事实。
4. 只有同一事实模型或同一生命周期的入口，才通过业务分类合并为“列表 -> 新建 -> 表单选择/默认业务分类”。
5. 台账、查询、报表、分析、配置和来源事实不并入办理表单。
6. 用户已确认的旧入口和锁定历史数据保留追溯价值，但不直接决定新系统菜单结构。

## 原始结论

应该先分析锁定基线中的用户可见列表，再完善正式办理能力。

原因是用户已经确认的不是单个页面，而是一套真实业务事实入口。后续完善如果绕开这些锁定入口，只在局部模型补按钮、补字段，就会反复出现“页面能打开但用户不知道怎么办理”“正式页面和验收面不一致”“入口越补越乱”的问题。

这些锁定入口是产品化办理能力的设计边界：

1. 用户可见菜单和列表数据保持锁定，不作为功能完善时的试验对象。
2. 历史事实数据只读，不写回、不覆盖。
3. 正式业务能力以统一办理口径承接，不被旧系统细碎入口牵着走。
4. 同一业务事实多入口展示时，保留用户确认入口，底层能力向统一正式模型收敛。
5. 来源事实入口只负责追溯，不作为新业务新增主入口。

## 需要合并承接的正式口径

### 1. 资金往来与账户调拨办理

承接入口：

- 承包人还项目款
- 承包人借项目款
- 项目借公司款登记
- 项目还公司款登记
- 账户间资金往来
- 资金日报表

产品口径：

- 公司与项目之间借还。
- 项目与项目之间调拨。
- 项目与承包人之间借还。
- 账户间转入转出。

落地要求：

- 办理单据必须具备来源项目、目标项目、往来对象、来源账户、目标账户、金额、方向、状态。
- 资金日报表只做分析汇总，不能替代办理入口。
- 历史锁定事实通过映射或派生视图进入资金总览，不回写历史事实表。

### 2. 付款与费用办理

承接入口：

- 支付申请
- 公司财务支出
- 往来单位付款
- 报销申请
- 项目费用报销单
- 扣款单
- 扣款实缴登记
- 扣款实缴退回
- 付款还保证金
- 承包人还项目款
- 项目还公司款登记
- 油卡登记
- 充值登记

产品口径：

- 付款申请 -> 付款执行 -> 费用/扣款/保证金/资金往来归集。
- 同一金额事实必须能解释项目、合同、往来单位、成本科目、资金账户。

落地要求：

- `payment.request`、`sc.payment.execution`、`sc.expense.claim` 建立清晰主链。
- 扣款、费用、保证金、项目还款不应该散落成互不相干的页面。
- 油卡和充值作为费用/资金来源明细追溯，不作为通用付款主入口。

### 3. 收入与收款办理

承接入口：

- 收入
- 工程进度款收入登记
- 到款确认表
- 自筹垫付收入
- 自筹垫付退回

产品口径：

- 项目收款登记是主办理入口。
- 到款确认、工程进度款、自筹垫付是来源类型或业务类型。

落地要求：

- 新增办理必须能选择项目、合同、往来单位、收款账户、收入类型。
- 到款确认和自筹事实保留只读追溯，通过映射进入项目收入汇总。

### 4. 合同与结算办理

承接入口：

- 一般合同（公司）
- 补充合同
- 收入合同执行
- 支出合同执行
- 收入合同结算
- 支出合同结算

产品口径：

- 合同是收款、付款、发票、结算、成本归集的业务主干。
- 结算不是附件展示页，必须承接合同、项目、往来单位和结算金额。

落地要求：

- 收入/支出合同各自有正式办理状态。
- 收入/支出结算可以新增、保存、确认，并回流合同执行。
- 合同台账能解释收款、付款、发票、结算来源。

### 5. 发票税务办理

承接入口：

- 进项发票
- 预缴税款
- 销项开票申请
- 销项开票登记
- 抵扣登记
- 外经证登记

产品口径：

- 发票税务要按项目、合同、往来单位归集。
- 抵扣登记是非现金税务事实，不能混成资金收支。
- 外经证作为税务来源明细追溯。

落地要求：

- 发票办理必须具备项目、合同、往来单位、发票类型、金额、税额、状态。
- 抵扣和预缴进入税务汇总，不改变现金口径。

### 6. 材料、库存、租赁与分包劳务机械

承接入口：

- 材料计划
- 报价单
- 入库单
- 出库单
- 租入
- 还租
- 分包方单
- 方单
- 零星用工
- 机械台班记录

产品口径：

- 材料采购库存形成计划、询价、入库、出库和成本归集链路。
- 租入/还租是租赁来源事实，不应替代租赁办理主入口。
- 劳务、零工、机械台班进入分包劳务机械办理口径。

落地要求：

- 材料、劳务、机械都必须回到项目、供应商/分包方、合同、成本科目。
- 出入库、劳务、机械发生额能进入项目成本。

### 7. 人事行政与资料证照

承接入口：

- 请假/休假审批单
- 公司人员名册
- 项目管理人员工资登记
- 社保人员登记
- 社保登记
- 补助
- 公司资料存档

产品口径：

- 人员、工资、社保、补助统一为人事行政办理。
- 公司人员名册是历史来源和人员主数据线索。
- 资料证照是公司/项目资料归档入口。

落地要求：

- 工资、社保、补助必须能关联项目、人员、期间。
- 公司人员名册不作为新业务办理主入口。

## 实施顺序

第一优先级是财务与合同主干，因为它决定项目经营事实是否能闭环：

1. 资金往来与账户调拨办理。
2. 付款与费用办理。
3. 收入与收款办理。
4. 合同与结算办理。
5. 发票税务办理。

第二优先级是成本来源：

1. 材料采购库存办理。
2. 分包劳务机械办理。
3. 施工日志与项目现场办理。

第三优先级是支撑型入口：

1. 人事行政办理。
2. 资料证照办理。
3. 配置入口权限收敛。

## 验收标准

后续每完成一个业务域，必须同时满足：

1. 用户确认的锁定菜单列表仍然通过现有基线门禁。
2. 该业务域的正式办理入口能新增、保存、提交/确认、查看附件。
3. 历史锁定事实只读可追溯，不被办理动作改写。
4. 新办理单据能关联项目、合同、往来单位、资金账户、成本科目中该业务域所需的关系。
5. 汇总/分析视图能解释来源明细和正式办理单据的关系。
6. 浏览器验收必须从用户可见菜单进入，不允许只验模型或后台 action。
