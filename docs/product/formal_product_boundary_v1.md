# 正式产品边界 v1

## 目标

本文件回答“平台、行业模块、用户模块、低代码配置到底各自应该交付什么正式产品”。后续涉及菜单、表单、列表、审批、用户偏好、初始化数据和配置沉淀时，必须先按本文判断归属，再进入实现。

正式产品不按“代码放在哪个目录方便”划分，而按事实权威划分：

- 平台产品提供通用机制。
- 行业产品提供可复制的行业标准。
- 用户产品提供特定客户的稳定交付基线。
- 低代码配置提供管理员运行时调整能力。
- 运维脚本只提供迁移、修复、重放和验证，不成为长期产品事实。

## 产品分层

| 层级 | 正式产品 | 当前载体 | 面向对象 | 事实权威 |
| --- | --- | --- | --- | --- |
| P0 | 平台内核产品 | `smart_core` | 平台研发、二开研发、所有行业模块 | 通用协议、契约、权限、配置引擎和运行时装配机制 |
| P1 | 施工行业标准产品 | `smart_construction_core` | 标准施工企业部署 | 行业模型、业务流程、菜单、默认页面、审批规则和行业字典 |
| P2 | 特定用户产品 | 仓外私有 `sce_customer_<tenant_key>` 模块 | 指定客户/租户 | 客户确认的菜单、表单、字段、角色、初始化数据和偏好 |
| P3 | 低代码配置产品 | 配置工作台 + `ui.business.config.contract` 等运行时表 | 客户管理员 | 管理员在界面显式保存的运行时配置 |
| P4 | 运维交付工具 | `scripts`、`docs/ops`、迁移脚本 | 交付、实施、研发 | 一次性修复、迁移、重放、验收和跨环境对齐 |

## 平台产品

平台产品交付“可复用机制”，不交付施工业务结论。

应提供：

- 登录、会话、租户/公司/用户上下文透传。
- intent envelope、handler 注册、API 返回结构、错误上下文。
- 页面契约、视图编排、菜单编排、版本、发布、回滚、快照和差异比较机制。
- 低代码配置引擎的通用能力：表单、列表、搜索、菜单、分析视图、审批配置入口的协议和持久化框架。
- 权限、审计、配置覆盖顺序、source authority、边界守卫。
- 前端只消费契约所需的通用渲染协议。

禁止提供：

- 施工行业字段、业务分类、状态流、审批岗位、菜单名称。
- 某个客户喜欢的三栏表单、不分 tab、字段顺序、菜单拆分。
- 依赖具体 action/model 名称的业务兜底。
- 通过前端猜测业务语义来修补后端契约。

## 行业标准产品

行业标准产品交付“可复制的施工行业默认系统”。

应提供：

- 施工行业核心模型、业务事实模型、投影模型、状态机和业务规则。
- 标准菜单、业务入口、action、默认列表/搜索/表单、默认审批策略。
- 行业字典、业务分类、标准角色/能力组、标准权限矩阵。
- 行业数据迁移和标准初始化。
- 行业验收矩阵：哪些业务入口必须可办、可查、可审批、可删除、可配置。
- 标准产品发版包和升级迁移。

禁止提供：

- 单个客户确认的表单顺序、字段隐藏、公司名称、个性化菜单。
- 为了某个客户验收而改变所有行业部署默认体验。
- 覆盖客户管理员通过低代码保存的运行时配置。
- 把临时数据修复沉淀成行业默认事实。

## 用户产品

用户产品交付“特定客户可重放的上线基线”。

应提供：

- 客户确认的菜单可见性、菜单拆分、表单三栏、不分 tab、字段顺序、字段显隐、默认标签。
- 客户特有角色、组织、公司、密码策略、安全策略、初始化参数。
- 客户数据字典和稳定数据补充，例如客户确认的地区、城市、分类补充。
- 客户确认的数据整理结果，例如稳定主数据补齐、默认组织/角色/公司映射、客户专属分类和可重放的数据基线。
- 已验收低代码配置的正式沉淀，使安装、升级、重建后仍能重放。
- 客户交付验收证据和回滚说明。

用户产品内部必须区分“功能/偏好”和“数据基线”：

| 子边界 | 承载内容 | 推荐载体 | 禁止事项 |
| --- | --- | --- | --- |
| 用户功能/偏好 | 表单布局、菜单拆分、字段顺序、字段显隐、标签、角色策略、客户安全策略 | 仓外私有 `sce_customer_<tenant_key>` 功能模块 | 混入一次性脏数据修复 |
| 用户数据基线 | 客户确认的主数据、字典、组织、公司、角色、城市/地区、数据整理沉淀结果 | 私有客户模块的 `data/`、客户数据模块或幂等 post-init 初始化 | 修改平台/行业默认语义；覆盖管理员运行时配置 |
| 用户数据迁移/修复 | 历史库清洗、一次性补锚、跨库对齐、人工确认残差处理 | P4 脚本、迁移 runbook、验收报告 | 长期作为页面、菜单、字段或业务事实来源 |

功能与数据必须分开审计、分开验收。用户功能/偏好证明界面可重放；用户数据基线证明数据可重放；一次性迁移修复证明执行过程可追踪。只有客户确认需要长期保留的数据整理结果，才能从 P4 运维工具沉淀到 P2 用户数据基线。

P2 用户模块发布前必须通过 `make verify.user_module.product_boundary`。该守卫检查用户数据基线、用户偏好和 post-init 重放入口是否分离，检查已确认低代码配置是否沉淀为 `lowcode_customer_config_baseline_manifest.v1`，并禁止表单偏好初始化隐藏执行数据字典创建或历史数据回填。

客户确认的真实用户主数据必须进入 P2 用户模块的数据基线载体。历史用户、客户专属字典、伙伴身份归类等稳定数据不能长期依赖 P4 脚本；但也不能作为普通 XML data 直接重复创建。用户模块必须使用幂等 loader：优先复用既有 XMLID，其次按业务唯一键复用，最后才创建，并且不得在重复安装/升级时复制同一用户、客户、供应商或归档无事实的新录入主数据。

客户历史业务数据基线的完整标准是“客户锁定的用户可见业务面”，不是某个迁移资产包目录的包数量。旧系统资产包、后续补锚修复、正式投影、用户可见菜单/表单/列表沉淀、审批附件追溯和运行态验收目标必须被同一个 P2 用户模块基线清单承载。该清单及其真实业务载荷只能位于私有客户包或受控私有存储，不进入标准产品仓库；产品仓仅保留通用接口、schema 和边界守卫。

完整历史业务载荷包含客户真实业务数据，不得进入公开 Git 仓库。P2 用户模块必须持有受控外部数据包锁 `docs/migration_alignment/migration_asset_package_lock_v1.json`、正式复放入口 `user_module.history_business_baseline.restore` 和运行态审计；数据包只能通过私有认证存储或本地受控路径提供。默认复放入口只 rehearsal 校验，必须显式设置 `USER_MODULE_HISTORY_BASELINE_APPLY=1` 才允许写库重放。

正式安装标准：

- 仅安装 P1 行业模块时，不得出现特定客户真实用户、客户专属主数据或客户历史数据基线；行业模块只提供通用行业模型、模板、菜单、权限和默认字典。
- 安装或升级 P2 用户模块时，客户确认的数据基线必须自动恢复，包括真实历史用户、客户专属字典、稳定主数据整理结果、历史业务可见面基线和用户偏好配置；大体量历史业务载荷通过用户模块声明的受控数据包锁和复放入口恢复。
- 用户模块数据基线必须幂等；同一数据库重复安装、升级或重放不得复制用户、客户、供应商，不得覆盖管理员后续显式配置，不得归档无事实的新录入主数据。
- P4 迁移脚本只能作为导入来源或一次性修复工具。完成验收并需要长期保留的数据，必须沉淀为 P2 用户模块的数据基线或进入 P3 低代码配置版本。

目标库安装或升级 P2 用户模块后，必须运行 `make verify.user_module.data_baseline.runtime_audit DB_NAME=<target_db>`。该验收命令重复重放真实用户数据基线，并确认用户数量、用户模块 XMLID 数量、登录名唯一性和历史业务可见面基线摘要保持稳定。

禁止提供：

- 平台通用契约合并机制。
- 行业标准流程和通用业务分类。
- 只在某次排查中临时需要的数据修复脚本。
- 冒充低代码即时配置，或覆盖管理员后续显式调整。

## 低代码配置产品

低代码配置产品交付“管理员可理解、可操作、可回滚的运行时配置能力”。

定位边界：

- 默认使用者是客户管理员、实施管理员和平台运营管理员，不是普通业务用户。
- 研发人员可以使用低代码配置做候选方案验证、运行时合同调试和配置生成，但开发态结果不能直接成为正式产品基线。
- 普通业务用户只能使用个人偏好能力，例如列表列宽、收藏筛选、快捷入口；不得修改全局菜单、字段、审批、权限或产品发布配置。
- 低代码配置是运行时覆盖层和编辑面，不是行业产品、用户产品或系统治理入口的事实权威。

低代码配置必须区分三类来源状态：

| 来源状态 | 使用者 | 正式含义 | 允许进入运行时 | 长期沉淀规则 |
| --- | --- | --- | --- | --- |
| `developer_draft` | 研发人员 | 候选配置、调试配置、生成草稿 | 仅限本地/测试/临时探针 | 必须转为代码、数据 XML、迁移脚本、产品策略或发布合同后才算正式 |
| `tenant_runtime` | 客户管理员、实施管理员 | 当前租户即时配置 | 可以进入当前租户运行时 | 客户确认长期生效后沉淀到 P2 用户产品，或保留为带版本审计的 P3 配置 |
| `product_release` | 产品发布流程 | 行业或平台正式发布基线 | 可以作为正式产品运行时基线 | 必须由 P0/P1/P2 的代码、数据、迁移或发布合同生成，不得只依赖手工低代码状态 |

菜单配置还必须区分三个对象边界：

| 菜单对象 | 定义 | 低代码允许动作 | 禁止动作 |
| --- | --- | --- | --- |
| 系统配置入口 | 配置中心、菜单配置、字段配置、权限/角色/策略配置、发布/合同/门禁治理入口 | 可改名、排序、归组；必须保留恢复路径和权限校验 | 不得被低代码彻底隐藏、删除或移动到不可达位置 |
| 产品发布菜单 | 行业模块正式能力入口和标准办理入口 | 可按租户做显示、命名、排序、分组覆盖；必须保留来源和版本 | 不得把临时客户偏好反向写成行业标准；不得让发布基线变成不可解释的运行时残留 |
| 用户/租户配置菜单 | 客户确认的菜单拆分、标签、分组、角色可见性 | 可作为 P3 即时配置，也可沉淀到 P2 用户模块 | 不得覆盖系统配置入口；不得修改业务事实本身 |

菜单运行时生效必须只有一个最终解释链路：

- Odoo 原生菜单和产品发布菜单只提供原始事实或产品基线。
- `ui.business.config.contract.menu_orchestration` 是低代码菜单运行时的优先覆盖源；读不到已发布合同才允许兼容回退到 `ui.menu.config.policy`。
- `release_navigation_v1.nav` 是产品发布基线和可配置目录的权威来源；它回答“默认交付哪些产品菜单”，不回答“当前用户最终看到什么”。
- `/api/menu/navigation` 或等价后端服务是运行态最终导航出口；它回答“当前用户此刻实际看到什么”，主导航必须消费这一最终解释结果。
- 菜单配置页必须拆成两个概念：可配置目录可以由产品发布基线、可恢复隐藏菜单和候选菜单组成；当前导航状态必须来自运行态最终导航解释。
- `policy.visible`、`menu_orchestration.visible` 只能表示管理员配置意图，不能直接翻译成当前隐藏。父级菜单即使配置为隐藏，只要存在可见子菜单，也必须作为运行态承载菜单显示。
- 前端不能用页面代码猜测“基础设置/配置中心”等业务语义，也不能用 `policy.visible` 自行推导运行态显示。主导航和菜单配置页的“当前显示状态”必须消费同一份后端运行态解释。
- 详细规则见 `docs/product/menu_configuration_runtime_boundary_v1.md`。

应提供：

- 菜单配置：显隐、命名、排序、移动、新增分组、复制入口、版本、回滚、生效检查。
- 表单配置：字段显隐、顺序、标签、新增字段、布局/分组、运行页面检查。
- 列表/搜索配置：列、排序、筛选、分组、版本和运行时生效检查。
- 审批配置：是否启用审批、审批方式、审批步骤拖拽、岗位/金额条件、运行态消费验证。
- 配置覆盖扫描、缺口补齐、快照导出、跨环境差异比较。
- 默认路径面向用户语言；高级治理信息只在高级设置中展示。

禁止提供：

- 业务事实修改，例如把一笔业务分类、金额、状态改成其他事实。
- 作为长期交付基线替代用户模块或行业模块。
- 作为开发人员正式发布机制的替代品；开发态低代码结果必须经过产品发布或模块沉淀。
- 作为普通业务用户修改全局系统配置、产品发布菜单、权限和审批流程的入口。
- 隐藏、删除或切断配置中心、菜单配置等系统配置恢复入口。
- 无版本、无审计、无作用域的隐式覆盖。
- 让前端承担“哪层配置覆盖哪层”的判断。

## 运维交付工具

运维工具只服务交付过程，不是产品事实权威。

允许：

- 一次性数据修复、历史迁移、跨库重放、开发服务器升级、验收脚本、差异报告。
- 生成候选配置或候选补丁，但必须沉淀到 P1/P2/P3 的正式载体后才能成为长期事实。

禁止：

- 用脚本长期决定页面字段顺序、菜单可见性或审批规则。
- 只在本地库修复，不进入可重放模块或低代码版本。
- 没有验收证据就宣称开发库与本地对齐。

## 归属判定规则

每次需求先按以下问题判定：

1. 是否所有行业都需要？是则属于平台产品；否则继续判断。
2. 是否所有施工行业标准部署都应该一致？是则属于行业标准产品。
3. 是否只是这个客户/租户/角色/用户确认的长期体验？是则属于用户产品。
4. 是否由客户管理员在界面即时调整，且需要版本、审计、回滚？是则属于低代码配置产品。
5. 是否只是迁移、修复、补数、验收？是则属于运维交付工具。

判定不清时，优先选择更靠近事实来源的一层：

- 机制靠平台。
- 行业语义靠行业模块。
- 客户确认靠用户模块。
- 管理员即时操作靠低代码运行时配置。
- 一次性动作靠运维脚本。

## 配置沉淀规则

低代码配置不是最终产品归属，它是编辑和运行时承载。

| 配置状态 | 正式归属 |
| --- | --- |
| 研发临时验证或生成的配置草稿 | `developer_draft`，不得作为正式产品事实 |
| 管理员临时试配 | P3 低代码运行时配置 |
| 管理员确认长期生效且只属于当前客户 | P2 用户产品 |
| 多个客户复用后成为施工标准默认 | P1 行业标准产品 |
| 配置引擎能力不足，需要新增通用能力 | P0 平台产品 |
| 修复历史脏数据或跨环境不一致 | P4 运维交付工具，完成后沉淀到 P1/P2/P3 |
| 客户确认的数据整理结果 | P4 可作为执行来源 | P2 用户数据基线 |

## 交付验收

正式产品交付不能只看代码合并，必须回答：

- 载体是否正确：代码模块、数据文件、低代码版本、脚本是否各在边界内。
- 功能与数据是否分开：用户功能/偏好、用户数据基线、一次性迁移修复是否各有独立载体和验收证据。
- 是否可重放：新库安装、模块升级、开发服务器升级后是否一致。
- 是否可覆盖：低代码运行时配置是否高于行业/用户默认配置。
- 是否可审计：source authority、版本、快照、差异是否可追踪。
- 是否可回滚：用户配置和发布版本是否能恢复。
- 是否经业务端验证：配置端成功后，业务页面必须验证生效。

## 当前结论

- `smart_core` 是平台产品，不再承载施工行业或客户偏好。
- `smart_construction_core` 是施工行业标准产品，承载标准业务对象和默认体验。
- P2 用户产品由仓外私有 `sce_customer_<tenant_key>` 模块承载；公共产品仓只定义通用交付接口与边界守卫。
- 低代码配置中心是平台提供的运行时产品能力，但它保存的是管理员显式配置，不替代行业模块或用户模块。
- 脚本和迁移只能作为交付工具，不能成为长期页面、菜单、审批和用户体验的事实来源。
