# 产品交付产品化审计 v1

- 分支：`codex/product-delivery-review`
- 审计日期：2026-05-23
- 审计视角：产品经理 / 交付负责人 / 客户成功
- 结论：当前系统具备较强的工程交付底座，但还没有完全形成客户可理解、可购买、可开通、可验收、可续费升级的产品闭环。

## 一、当前优势

### 1. 平台底座已经具备产品化前提

系统已经形成以 `smart_core` 为平台底座、行业模块承载业务能力的架构边界。`ui.contract`、场景契约、能力目录、发布快照、产品策略、权限治理、前端通用渲染等机制，为后续多行业、多版本、多租户交付提供了基础。

### 2. 产品能力映射已经初具规模

`docs/product/product_capability_matrix_v2.md` 已经把 63 个 capability 映射到 20 个产品能力和 10 个行业模块，说明能力抽象不是临时堆功能，而是已经具备产品分层的雏形。

### 3. 交付治理比普通项目更成熟

仓库中已有 delivery playbook、delivery package manifest、readiness scoreboard、生产部署 runbook、发布控制台、角色矩阵和多类 guard。工程交付、可重放、可审计的意识较强。

### 4. 角色与场景已经开始从技术菜单转向业务入口

已有 `user_journey_pm.md`、`user_journey_finance.md`、`user_journey_exec.md`、`roles_v1.md`、`role_capability_profiles.md` 等材料，说明项目已经开始关注“用户以什么身份完成什么任务”。

## 二、核心问题

### 1. 产品包还没有真正成型

`smart_construction_bundle` 当前仍是薄 bundle，依赖 `smart_core` 且 `data` 为空。它还不能代表一个可交付的施工行业产品包。

需要补齐：

- 产品包包含哪些菜单、角色、默认策略、场景和验收资产。
- 标准版、专业版、企业版分别开放哪些能力。
- 行业产品包与平台能力、客户公司授权、许可证之间如何绑定。
- 安装产品包后，客户看到的第一个业务可用状态是什么。

### 2. 功能索引与产品能力矩阵不一致

能力矩阵已经很丰富，但 `feature_index.zh.md` 仍然是轻量功能列表，只覆盖少数稳定功能。客户、销售、实施、培训无法仅通过功能索引理解产品边界。

需要把功能索引升级为产品能力目录：

- 面向客户语言，而不是模块语言。
- 每项能力明确目标角色、业务价值、入口、依赖、可用版本、验收方式。
- 标识能力状态：已交付、试用、预览、内部治理、仅演示。

### 3. 开通路径仍偏工程操作

`delivery_playbook_v1.md` 主要描述安装模块、配置扩展、准备用户、跑验证命令。它适合工程实施，不适合客户交付。

需要转为开通旅程：

- 创建企业与组织。
- 选择行业产品包和版本。
- 配置业务角色与权限模板。
- 初始化基础字典、税率、项目类型、审批流程。
- 选择空白启用、演示数据、历史数据迁移三种模式。
- 交付前置检查、失败恢复、验收签字。

### 4. 验收证据偏工程，客户不可读

`product_delivery_gap_report.md` 可以显示 `gap_count=0`，但这只能证明仓库中若干工程文件齐全。它不能证明客户能顺利完成业务，也不能证明产品值得购买。

客户可读证据包应该包含：

- 本次交付范围。
- 已开放菜单与角色。
- 关键业务路径截图或浏览器证据。
- 数据初始化与迁移结果。
- 已知限制。
- 回滚和支持方式。
- 客户验收清单。

### 5. 首次使用体验仍缺少产品引导

登录页已有价值主张，但系统登录后的产品首跳仍主要依赖角色、菜单和契约渲染。需要把“用户今天要完成什么工作”作为主线，而不是让用户自己理解菜单体系。

应补齐：

- 我的工作作为统一首屏。
- 每个角色的默认任务中心。
- 空状态引导。
- 新用户首次登录引导。
- 关键业务路径的下一步建议。

### 6. 配置管理员能力需要治理化

业务配置管理员拥有较强权力，但产品形态不应只是“能改配置”。应形成受控配置发布机制：

- 草稿配置。
- 影响预览。
- 审批发布。
- 回滚。
- 变更审计。
- 模板化初始化。

### 7. 商业化与授权链路还不完整

`smart_license_core` 已经存在，`product_tier_policy_v1.json` 也有 community/pro/enterprise 分层，但产品包、能力策略、授权、客户公司、版本快照之间还需要形成统一可解释链路。

客户和实施都需要看到：

- 当前购买版本。
- 当前开放能力。
- 超出版本的能力为什么不可用。
- 如何升级。
- 授权到期和用量限制。

### 8. 支持与运营能力不足

生产 runbook 完整，但系统内的支持能力需要产品化：

- 当前版本与发布快照。
- 系统健康检查。
- 业务数据健康度。
- 诊断包导出。
- 最近配置变更。
- 最近发布与回滚记录。
- 支持人员只读诊断入口。

## 三、建议产品架构

建议将产品交付拆成六层：

1. 产品包：施工企业管理系统、业主方管理系统等行业产品。
2. 版本套餐：community / pro / enterprise，或 standard / pro / enterprise。
3. 角色旅程：项目经理、财务、经营人员、配置管理员、领导。
4. 开通交付：企业、组织、权限、字典、审批、项目、历史数据。
5. 验收证据：角色路径、数据口径、浏览器证据、缺口清单。
6. 运营支持：健康、授权、发布、诊断、审计、升级。

## 四、第一轮优先级

### P0：先把产品包交付闭环补成

- `smart_construction_bundle` 从空壳变为真实产品包。
- 定义施工行业标准产品包的菜单、角色、策略、默认入口、验收资产。
- 产品能力目录替代轻量功能索引。
- 标准版/专业版/企业版能力边界可解释。
- 输出客户可读交付证据包模板。

### P1：补首次开通和角色旅程

- 企业开通向导设计。
- 角色模板与默认首页。
- 五条黄金路径：项目经理、财务、经营、配置管理员、领导。
- 空状态与下一步动作。
- 演示数据重置与销售演示脚本。

### P2：补运营、授权与续费升级

- 授权状态与能力不可用解释。
- 发布快照对比。
- 配置变更审计与回滚。
- 支持诊断包。
- 产品使用数据与客户成功指标。

## 五、验收口径

一项能力只有同时满足以下条件，才算产品化完成：

- 客户知道它解决什么问题。
- 目标角色知道从哪里进入。
- 系统知道它属于哪个产品包和哪个版本。
- 管理员知道如何开通、关闭、授权和回滚。
- 实施知道如何初始化和验收。
- 支持知道如何诊断问题。
- 工程有自动化检查证明上述材料没有漂移。

## 六、本轮建议落地物

本轮先落地两个资产：

- 本审计文档：`docs/product/product_delivery_productization_audit_v1.md`
- 产品化 readiness 检查：`scripts/verify/product_delivery_productization_readiness.py`

后续再进入代码实现，不建议直接继续加功能。当前最重要的是先把“产品包、版本、角色、开通、验收、支持”这六条线固定成可验证结构。
