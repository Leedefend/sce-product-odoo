# 正式办理能力闭环迭代计划 V1

## 目标

本轮迭代从“产品菜单发布完成”进入“正式办理能力闭环”。验收目标是：用户历史数据进入正式产品模型后，可以像新办理业务一样正常查看、编辑、新增、审批、追溯；低代码配置可以覆盖用户体验，但不能破坏产品必需办理能力。

## 边界

- 行业模块负责正式模型、字段、关系、状态、审批、附件和办理动作。
- 用户模块负责历史数据整理、确认、回填和来源追溯，不承载正式产品办理逻辑。
- 低代码配置负责运行态体验调整，不拥有产品能力定义权，也不能隐藏系统配置入口或删除产品必需字段。
- 发布菜单是入口，不等于办理闭环；办理闭环必须覆盖运行态合同、CRUD 权限、字段承载、关系、审批和附件。

## 第一阶段：能力矩阵

以 `docs/product/full_product_capability_scope_v1.md` 中 81 个已发布正式办理能力为基准，建立可执行矩阵：

- 菜单是否存在且启用。
- 菜单是否绑定 `ir.actions.act_window`。
- 动作模型是否存在。
- 是否具备 read/create/write 基础权限。
- 是否有 list/tree、form、search 运行态合同。
- list/form 运行态是否能生成字段结构。
- form 运行态是否保留无默认值的 ORM 必填字段，低代码配置不得隐藏正式办理必填项。
- 是否能形成可访问办理 URL。

门禁：`make verify.formal_business_operation_capability_matrix`

## 第二阶段：高频流程深验收

优先深挖 8-12 条高频流程：

- 项目台账/项目立项
- 一般合同、施工合同
- 材料计划、材料入库、材料出库
- 分包申请、分包结算
- 劳务用工、机械台班
- 收款登记、付款申请、费用报销
- 发票税务：销项开票、进项发票、抵扣登记、外经证

每条流程验收：

- 历史数据打开可读。
- 正式字段可编辑并保存。
- 可以新增同类业务。
- 关系字段可选、可保存、可追溯。
- 审批/提交状态不阻断正常办理。
- 附件入口可见且来源追溯不丢失。

门禁：`make verify.formal_business_operation_core_flow_smoke`

第一批 smoke 样本覆盖：一般合同、施工合同、材料计划、材料入库、材料出库、分包申请、分包结算、方单、机械台班、收款登记、费用报销、进项发票。该门禁以正式菜单动作上下文执行 ORM 新增和编辑，并在事务内回滚，不整理、不改写用户历史数据。

## 第三阶段：低代码覆盖稳定性

- 用户配置必须最终生效。
- 产品必需字段不能被用户配置破坏。
- 系统配置入口不能被运行态低代码隐藏。
- 配置页、办理页、列表页、搜索页和菜单导航必须一致。

## 收口标准

- 能力矩阵 81/81 通过，缺口为 0。
- 高频流程深验收形成可重复门禁。
- 本地与日常开发服务器均通过完整发布门禁。
- 门禁生成物不会让 Git 工作区变脏。
