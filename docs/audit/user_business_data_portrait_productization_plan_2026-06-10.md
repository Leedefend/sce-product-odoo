# 用户真实业务数据画像与产品化承接计划 - 2026-06-10

环境：本地开发库 `sc_demo`

画像脚本：`scripts/verify/user_business_data_portrait.py`

画像产物：`artifacts/user_business_data_portrait.sc_demo.json`

## 结论

本轮不能再按“旧入口/旧菜单”推进业务整合。当前用户历史数据已经覆盖项目主数据、合同结算、收入收款、付款费用、税务发票、账户往来、材料采购库存、分包劳务机械、现场进度质量安全、预算成本、审批附件治理 11 个能力域。旧系统的入口粒度只是历史操作习惯，不等于新系统的产品化办理口径。

正式产品应采用“业务办理入口 + 来源事实明细 + 结果汇总视图”的结构：

- 业务办理入口：用户新增、审批、确认、完成业务的主入口。
- 来源事实明细：承接历史迁移数据、解释汇总结果的只读或半只读明细。
- 结果汇总视图：项目、往来单位、合同、账户、成本、税务等经营口径的分析视图。

后续功能完善必须保持用户已验收的菜单、列表可见面和历史业务事实数据稳定；新增能力应进入产品化业务入口，不能为了单点功能随意改动用户确认过的菜单体系，也不能改写已锁定事实数据。

## 产品化业务口径

### 1. 项目经营主线

统一承载对象：项目、公司、客户、供应商、合同、账户、成本科目。

产品入口：

- 项目台账
- 客户/供应商
- 资金账户
- 成本科目
- 业务主体

关键要求：

- 新发生的业务单据必须具备项目关系口径；能确定往来单位、合同、账户的，必须在新办理单据或非侵入式映射层承载。
- 历史名称字段只能作为追溯证据，不能替代正式主数据关系。

### 2. 合同与结算主线

统一承载对象：收入合同、支出合同、结算单、合同台账、付款/收款来源。

产品入口：

- 收入合同办理
- 支出合同办理
- 收入合同结算
- 支出合同结算
- 合同执行台账

关键要求：

- 合同不只是列表展示，必须能驱动收款、付款、发票、结算和成本。
- 当前 `sc.settlement.order` 存在较多合同关系缺口，需要形成合同归集口径专项；历史锁定事实只读使用，不写回。

### 3. 收入与收款主线

统一承载对象：工程进度收款、到款确认、自筹收入、自筹退回、收款发票关系。

产品入口：

- 项目收款登记
- 工程进度收款
- 自筹资金收入/退回
- 收款发票核销

关键要求：

- 不再保留“到款确认表”“自筹收入”等旧入口作为主办理入口，而是作为来源明细。
- `sc.receipt.income` 应成为正式办理主模型；历史收款事实只读追溯，新发生收款必须具备往来单位、合同、付款/收款申请关系。

### 4. 付款与费用主线

统一承载对象：付款申请、付款执行、费用报销、扣款实缴/退回、保证金支付/退回、项目还公司款。

产品入口：

- 付款申请
- 付款执行
- 费用与保证金单据
- 扣款/退回办理

关键要求：

- `payment.request` 和 `sc.payment.execution` 是付款办理主链。
- `sc.expense.claim` 内部 `claim_type` 已经承接多类旧事实，但需要进一步产品化为清晰的业务类型入口，避免用户看到“费用与保证金单据”后仍不理解该办什么。

### 5. 税务与发票主线

统一承载对象：进项发票、销项发票、抵扣登记、扣款税务事实、发票成本进度。

产品入口：

- 发票登记
- 抵扣登记
- 销项发票台账
- 进项发票/成本进度分析

关键要求：

- `sc.invoice.registration` 与 `sc.tax.deduction.registration` 需要统一合同、项目、往来单位口径。
- 抵扣登记不是资金收支，应作为非现金税务事实进入项目经营口径。

### 6. 账户与往来资金主线

统一承载对象：公司借项目款、项目还公司款、承包人借项目款、承包人还项目款、项目间/账户间调拨、资金日报。

产品入口：

- 资金往来办理
- 账户调拨
- 融资与借款登记
- 项目资金总览
- 往来对象资金总览

关键要求：

- 汇总视图只做经营分析，不作为用户办理入口。
- `sc.fund.account.operation` 和 `sc.financing.loan` 应统一为“资金往来/账户调拨”办理口径，用户不应被迫理解旧表来源。
- 现有关系缺口集中在账户、往来单位、来源项目，需要按业务事实设计非侵入式分类和新办理必填规则。

### 7. 材料采购库存主线

统一承载对象：材料目录、材料计划、询价、采购申请、验收、入库、出库、材料结算。

产品入口：

- 材料计划
- 材料采购申请
- 材料询价
- 材料验收
- 入库/出库
- 材料结算

关键要求：

- 材料目录体量极大，必须先做目录治理、重复归并、项目适用范围和价格口径。
- 材料办理链应与成本台账、合同、付款申请打通。

### 8. 分包劳务机械主线

统一承载对象：分包计划/申请/登记/结算、劳务计划/申请/考勤/使用/结算、机械计划/申请/使用/结算。

产品入口：

- 分包办理
- 劳务办理
- 机械办理

关键要求：

- 当前有大量模型具备办理动作，但真实数据集中在机械使用、劳务使用、少量分包申请。
- 下一步不应做大而散的入口，应先围绕“使用记录 -> 结算 -> 成本 -> 付款”的新办理闭环完善。

### 9. 现场进度质量安全主线

统一承载对象：施工日志、进度、任务、现场证据、质量问题、安全问题、整改复查。

产品入口：

- 施工日志
- 进度登记
- 质量问题整改
- 安全问题整改
- 现场证据

关键要求：

- `project.task` 和现场证据数据量大，不能只看财务菜单。
- 现场数据应服务项目履约、结算依据、风险追溯，而不是孤立列表。

### 10. 预算成本管控主线

统一承载对象：目标成本、预算清单、成本科目、成本分摊、成本台账、成本期间。

产品入口：

- 目标成本
- 成本科目
- 成本台账
- 成本期间锁定
- 综合成本分析

关键要求：

- 成本台账已经有 73,068 条，是判断系统是否能真正办理业务的核心之一。
- 付款、材料、分包、劳务、机械、发票都应最终沉淀到成本口径。

### 11. 审批附件与治理主线

统一承载对象：附件、审批策略、审批步骤、历史待办、历史流程、审计日志。

产品入口：

- 审批配置
- 附件证据
- 历史流程追溯
- 审计日志

关键要求：

- 附件和历史流程不是辅助信息，是业务可用性的硬前提。
- 所有正式办理动作必须保留附件、审批、操作日志的追溯链。

## 优先级

P0：稳定用户已验收可见面

- 菜单、列表字段、历史数据可见面继续硬拦截。
- 新能力开发不得改变用户确认过的菜单体系和字段显示。

P1：建立统一业务办理入口

- 项目收款登记
- 付款申请/付款执行
- 发票登记/抵扣登记
- 资金往来/账户调拨
- 合同结算
- 成本台账

P2：建立关系口径和非侵入式事实归集

- 收款建立往来单位、合同、收款申请的归集口径。
- 付款建立合同、付款申请、往来单位的归集口径。
- 资金往来建立账户、项目、往来对象的归集口径。
- 发票建立合同、项目、往来单位的归集口径。
- 结算建立合同、往来单位的归集口径。
- 以上均通过新办理链路、派生视图或映射层承载，不改写用户已锁定事实。

P3：打通业务闭环

- 合同 -> 结算 -> 收款/付款 -> 发票 -> 成本 -> 资金总览。
- 材料/分包/劳务/机械 -> 验收/使用 -> 结算 -> 成本 -> 付款。
- 现场进度/任务/证据 -> 合同履约 -> 结算依据。

P4：沉淀为产品化能力

- 对旧系统来源做迁移解释和追溯，不把旧表名暴露给用户。
- 对新系统办理入口使用业务语言，不使用技术模型名。
- 对不同用户复用同一套能力域和办理流，只在菜单策略、字段扩展、审批策略上做配置。
