# SCEMS v1.0 发布缺口分析（Release Gap Analysis）

## 1. 分析目标
识别当前系统距离 `SCEMS v1.0` 首发标准的关键缺口，并给出可执行补齐路径。

## 2. 发布标准对照

### 2.1 目标标准
- 可部署
- 可演示
- 可培训
- 可实际使用

### 2.2 核心业务路径
- 登录
- 我的工作
- 项目台账
- 项目驾驶舱
- 合同/成本/资金查看
- 风险识别

## 3. 现状与缺口

### A. 产品与场景
- 现状：核心 Scene 与 delivery policy 已具备基础能力
- 缺口：`project.management` 的 7-block 业务语义需进一步与真实业务数据对齐

### B. 服务与能力
- 现状：contract/scene 编排能力较完整
- 缺口：`ProjectDashboardService` 等聚合服务需形成稳定接口与指标口径定义

### C. 权限与角色
- 现状：已有 prod-like 角色验证基础
- 缺口：V1 角色矩阵与 block 可见性规则需形成明确配置清单

### D. 验证体系
- 现状：已有 `phase_next.evidence.bundle`、scene governance 等验证链路
- 缺口：缺少 V1 发布视角的汇总验证入口（dashboard/route/permission 专项）

### E. 部署与运维
- 现状：有 Makefile 与容器化基础
- 缺口：缺少 V1 面向交付的部署指南与回滚演练文档

### F. 演示与验收
- 现状：已有部分演示/发布文档基础
- 缺口：缺少 V1 专属演示脚本与用户验收清单

## 4. 缺口优先级

### P0（首发阻断）
- 发布范围冻结与变更机制
- 7-block 控制台语义与数据口径闭环
- 角色权限矩阵落地
- 部署指南/演示脚本/验收清单齐备

### P1（首发增强）
- V1 验证入口聚合与报告可视化
- 培训材料与标准演示脚本模板化

### P2（后续优化）
- 指标体系扩展
- 场景个性化与体验优化

## 5. 补齐计划（Phase 对齐）
- Phase 0：范围冻结 + 资产盘点 + 缺口分析（本文件）
- Phase 1：导航收口与 delivery policy 固化
- Phase 2：核心场景闭环（4 大场景）
- Phase 3：角色权限体系
- Phase 4：前端体验稳定
- Phase 5：验证与部署资料齐套
- Phase 6：试运行与首发

## 6. 退出条件（Definition of Done）
- V1 范围内能力全部通过发布级验证
- 核心业务路径在 prod-like 角色下可复现
- 发布文档齐备并完成评审
- 首发演示可由非研发人员按脚本独立完成
