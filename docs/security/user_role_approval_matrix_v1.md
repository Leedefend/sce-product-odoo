# 用户角色组与审批岗位矩阵 v1

## 目标

用户配置管理员只分配业务角色组。能力组仍由平台管理员维护，作为角色组背后的权限底座。审批规则使用“审批岗位”表达真实业务流转，不直接让用户理解技术能力组。

本轮先解决用户维护表中的实际岗位能否承载业务和审批，不一次性写死所有审批流。

## 分层边界

| 层级 | 配置对象 | 维护人 | 用途 |
| --- | --- | --- | --- |
| 能力组 | `SC 能力 - ...` | 平台管理员 | ACL、菜单、模型权限、底层审批能力 |
| 业务角色组 | `SC 角色 - ...` | 用户配置管理员 | 给用户分配日常岗位和业务职责 |
| 审批岗位 | 审批规则步骤里的岗位 | 业务配置管理员 | 表达谁初审、复审、终审 |
| 审批规则 | 单据、金额、步骤、执行组 | 业务配置管理员 | 控制不同业务单据的审批链 |

## 首批审批流建议

| 业务单据 | 建议步骤 |
| --- | --- |
| 施工/收入合同 | 合同经办人 -> 经营审核人 -> 项目负责人 -> 成控审核人 -> 管理层/总经理终审 |
| 一般合同 | 合同经办人 -> 经营审核人 -> 财务审核人 -> 管理层/总经理终审 |
| 采购订单 | 项目部/工程部经办 -> 项目负责人 -> 采购审核人 -> 财务审核人 |
| 付款申请 | 财务经办人 -> 项目负责人 -> 财务审核人 -> 管理层/总经理终审 |
| 收款登记 | 财务经办人 -> 经营审核人 -> 财务审核人 |
| 结算单 | 结算经办人 -> 项目负责人 -> 成控审核人 -> 结算审核人 |
| 客户/供应商资料 | 客商资料审核人 -> 业务配置管理员 |

## 设计约束

- 管理层/总经理不 implied `base.group_system`，不能变成平台管理员。
- 管理员角色拆为业务配置管理员和平台管理员；用户配置管理员只分配业务角色组。
- 受限财务默认只读，不进入默认审批终审。
- 旧审批历史不写入 `tier.review`，历史审批继续作为只读审计资产；新系统审批只通过正常提交生成 runtime review。
- 角色组名称贴近用户可见岗位，能力组名称留在平台维护层。

## 验证入口

角色组、审批岗位和默认审批链由以下守护脚本校验：

```bash
make verify.user_role_approval_matrix.guard DB_NAME=<db>
```

该脚本检查：

- 用户配置管理员可分配的必须全部是业务角色组，不能泄露能力组或基础组。
- 17 个审批岗位必须能解析到预期角色组或能力组。
- 施工/收入合同、一般合同、采购订单、结算单、付款/收款申请必须是用户可见多级审批模板。
- 收款登记、结算调整默认不强制审核，但保留建议步骤模板。
